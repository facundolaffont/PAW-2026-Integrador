Diagramas de Secuencia

Flujos principales de UNO Argentino: autenticación JWT, gestión de salas vía REST, partida en tiempo real vía WebSocket y persistencia de puntajes en MySQL.

HTTP / REST WebSocket Polling (3 s)

1. Registro de usuario REST

Un visitante crea una cuenta. El servidor valida datos, hashea la contraseña con bcrypt y almacena el jugador en MySQL. Emite un JWT en cookie httpOnly. 

sequenceDiagram
    actor U as Usuario
    participant B as Browser
    participant MA as ManejadorAuth
    participant AC as AuthController
    participant P as Persistencia
    participant DB as MySQL

    U->>B: Abre /public/registrarse
    B->>MA: GET /api/me
    MA-->>B: 401 (no autenticado)
    U->>B: Completa formulario y envía
    B->>MA: POST /api/registrarse {nombreUsuario, password}
    MA->>AC: registrar(nombre, password)
    AC->>AC: Validar nombre y contraseña
    AC->>P: registrarJugador(usuario)
    P->>DB: INSERT jugador
    DB-->>P: jugadorId
    P-->>AC: jugador creado
    AC-->>MA: {jugadorId, nombreUsuario}
    MA->>MA: Firmar JWT
    MA-->>B: Set-Cookie: token (httpOnly)
    B-->>U: Redirect /public/

Archivos: registro.ejs, manejadorAuth.js, AuthController.js

2. Login y logout REST

Autenticación con credenciales existentes. Las páginas protegidas usan requireAuthWeb para redirigir si el JWT es inválido o expiró. 

sequenceDiagram
    actor U as Usuario
    participant B as Browser
    participant MA as ManejadorAuth
    participant AC as AuthController
    participant P as Persistencia
    participant MW as middlewareAuth

    rect rgb(30, 40, 55)
        Note over U,MW: Login
        U->>B: Abre /public/ingresar
        U->>B: Envía credenciales
        B->>MA: POST /api/ingresar {nombreUsuario, password}
        MA->>AC: ingresar(nombre, password)
        AC->>P: buscarPorNombre(nombre)
        P-->>AC: jugador + hash
        AC->>AC: bcrypt.compare(password, hash)
        AC-->>MA: {jugadorId, nombreUsuario}
        MA-->>B: Set-Cookie: token
        B-->>U: Redirect /public/
    end

    rect rgb(30, 40, 55)
        Note over U,MW: Acceso a página protegida
        U->>B: Navega /public/jugar
        B->>MW: GET /public/jugar (cookie token)
        MW->>MW: Verificar JWT
        alt Token válido
            MW-->>B: Renderizar página
        else Token inválido / ausente
            MW-->>B: Redirect /public/bienvenida
        end
    end

    rect rgb(30, 40, 55)
        Note over U,MW: Logout
        U->>B: Clic "Salir"
        B->>MA: POST /api/salir
        MA-->>B: Clear-Cookie: token
        B-->>U: Redirect /public/bienvenida
    end

Archivos: login.ejs, middlewareAuth.js, manejadorAuth.js

3. Crear sala de juego REST

El creador elige cantidad de jugadores (2–4) y bots (0–3). La sala se guarda en memoria con el creador y los bots ya asignados. 

sequenceDiagram
    actor U as Usuario
    participant B as Browser
    participant MP as ManejadorPartidas
    participant PC as PartidaController
    participant S as SalaDeJuego
    participant P as Persistencia

    U->>B: /public/jugar → /public/crear-sala
    U->>B: Configura maxJugadores y cantidadBots
    B->>MP: POST /api/partidas {maxJugadores, cantidadBots}
    Note over MP: requireAuth → req.jugadorId
    MP->>PC: crearPartida(jugadorId, opts)
    PC->>P: buscarJugador(jugadorId)
    P-->>PC: datos jugador
    PC->>PC: Validar límites y que no esté en otra partida
    PC->>S: new SalaDeJuego(creador, bots)
    S-->>PC: sala inicializada (estado: esperando)
    PC->>P: guardarPartida(partidaId, sala)
    P-->>PC: OK (memoria)
    PC-->>MP: {partidaId, resumen}
    MP-->>B: 201 Created
    B-->>U: Redirect /public/partida?partidaId=X

Archivos: crear-sala.ejs, manejadorPartidas.js, PartidaController.js, SalaDeJuego.js

4. Buscar y unirse a sala REST + Polling

El listado de salas disponibles se refresca cada 3 segundos. Al unirse, el servidor valida que la sala exista y tenga lugar antes de renderizar la partida. 

sequenceDiagram
    actor U as Usuario
    participant B as Browser
    participant MP as ManejadorPartidas
    participant PC as PartidaController
    participant MF as ManejadorFront
    participant P as Persistencia

    U->>B: Abre /public/salas
    loop Cada 3 segundos
        B->>MP: GET /api/partidas
        MP->>PC: listarPartidas()
        PC->>P: listarPartidasDisponibles()
        P-->>PC: salas con estado "esperando"
        PC-->>MP: [{partidaId, jugadores, max...}]
        MP-->>B: JSON lista de salas
        B->>B: Actualizar tabla HTML
    end

    U->>B: Clic "UNIRSE" en una sala
    B->>MF: GET /public/partida?partidaId=X
    MF->>PC: obtenerPartida(id, jugadorId)
    PC->>P: obtenerPartida(id)
    P-->>PC: sala

    alt Sala llena o no existe
        PC-->>MF: error 403/404
        MF-->>B: Redirect /public/salas?error=...
    else Hay lugar
        PC-->>MF: resumen partida
        MF-->>B: Render partida.ejs
        B-->>U: Pantalla de lobby
    end

Archivos: salas.ejs, manejadorFront.js, PartidaController.js

5. Lobby, conexión WS e inicio de partida WebSocket

Tras cargar la página, el cliente abre WebSocket autenticado por cookie JWT. El creador inicia la partida cuando hay suficientes jugadores conectados. 

sequenceDiagram
    actor U as Usuario
    participant B as Browser (partida.js)
    participant App as app.js (WS)
    participant MM as ManejadorMensajes
    participant PC as PartidaController
    participant MC as ManejadorConexiones
    participant S as SalaDeJuego
    participant Otros as Otros jugadores

    B->>B: GET /api/me → jugadorId
    B->>PC: GET /api/partidas/:id (resumen lobby)
    PC-->>B: estado, jugadores, bots

    B->>App: WebSocket /ws?partidaId=X (cookie JWT)
    App->>App: Verificar JWT en upgrade
    App->>MM: manejarConexion(ws, jugadorId, partidaId)
    MM->>MC: registrar(jugadorId, ws)
    MM->>PC: unirJugador(partidaId, jugadorId)
    PC->>S: agregarJugador / reconectar
    PC->>MC: emitirATodos("jugador-unido")
    MC-->>Otros: {evento: jugador-unido}
    PC->>MC: emitirA(jugadorId, "estado-partida")
    MC-->>B: {evento: estado-partida, datos: {...}}

    Note over U,S: Creador inicia cuando la sala está lista
    U->>B: Clic "INICIAR PARTIDA"
    B->>MM: {accion: iniciar-partida}
    MM->>PC: iniciarPartida(partidaId, jugadorId)
    PC->>S: iniciar() — repartir cartas
    S-->>PC: estado = jugando
    PC->>MC: emitirATodos("estado-partida")
    PC->>MC: emitirATodos("turno-cambiado")
    MC-->>B: estado inicial del juego
    MC-->>Otros: estado inicial del juego

    opt Turno de un bot
        PC->>PC: ejecutarTurnoBot()
        PC->>MC: emitirATodos(carta-jugada, turno-cambiado)
    end

Archivos: partida.js, app.js, manejadorMensajes.js, PartidaController.js

6. Jugar un turno (humano) WebSocket

El jugador en turno juega una carta o roba del mazo. El servidor valida reglas UNO, penalidades y transmite el nuevo estado a todos los clientes. 

sequenceDiagram
    actor U as Jugador en turno
    participant B as Browser (partida.js)
    participant MM as ManejadorMensajes
    participant PC as PartidaController
    participant S as SalaDeJuego
    participant MC as ManejadorConexiones
    participant Otros as Otros clientes

    alt Jugar carta
        U->>B: Clic en carta de la mano
        opt Carta comodín (+4 / cambio color)
            B->>B: Modal elegir color
        end
        B->>MM: {accion: jugar-carta, cartaId, colorElegido?}
        MM->>PC: jugarCarta(partidaId, jugadorId, cartaId, color)
    else Robar carta
        U->>B: Clic "Robar"
        B->>MM: {accion: robar-carta}
        MM->>PC: robarCarta(partidaId, jugadorId)
    end

    PC->>S: jugarCarta / robarCarta
    S->>S: Validar turno, carta legal, UNO, penalidades
    S-->>PC: resultado {ok, eventos...}

    PC->>MC: emitirATodos("carta-jugada")
    MC-->>Otros: animación carta en mesa

    opt Jugador robó
        PC->>MC: emitirA(jugadorId, "cartas-robadas")
        MC-->>B: cartas nuevas (solo al que robó)
    end

    PC->>MC: emitirATodos("turno-cambiado")
    PC->>MC: emitirATodos("estado-partida")
    MC-->>B: UI actualizada
    MC-->>Otros: UI actualizada

    opt Siguiente turno es bot
        PC->>PC: ejecutarTurnoBot()
        Note over PC: BotLLM.decidirJugada (Gemini)
        PC->>MC: emitirATodos(eventos del bot)
    end

Archivos: partida.js, SalaDeJuego.js, BotLLM.js, manejadorMensajes.js

7. Fin de ronda y fin de partida WebSocket

Al vaciar la mano gana la ronda. Se acumulan puntos hasta 200. Al terminar la partida se persisten deltas en MySQL y se elimina la sala de memoria. 

sequenceDiagram
    actor U as Jugador
    participant B as Browser (partida.js)
    participant PC as PartidaController
    participant S as SalaDeJuego
    participant MC as ManejadorConexiones
    participant P as Persistencia
    participant DB as MySQL

    Note over S: Jugador juega su última carta
    S->>S: Detectar mano vacía
    S->>S: estado = entre-rondas, calcular puntos ronda
    PC->>MC: emitirATodos("ronda-terminada")
    MC-->>B: {ganadorRonda, puntosGanados, puntajesRonda}
    B->>B: Modal "Fin de ronda"

    U->>B: Clic "Continuar"
    B->>PC: {accion: continuar-ronda}
    PC->>S: continuarRonda()

    alt Nadie llegó a 200 pts
        S->>S: Repartir nueva ronda
        S-->>PC: estado = jugando
        PC->>MC: emitirATodos("estado-partida")
        PC->>MC: emitirATodos("turno-cambiado")
    else Alguien ≥ 200 pts — partida terminada
        S->>S: partidaTerminada = true
        PC->>P: guardarResultadoPartida(ranking, deltas)
        P->>DB: INSERT partida + partida_jugadores
        DB-->>P: OK
        PC->>P: eliminarPartida(partidaId)
        PC->>MC: emitirATodos("partida-terminada")
        MC-->>B: Modal ranking final + deltas globales
        B->>B: Opción volver al inicio
    end

Archivos: SalaDeJuego.js, PartidaController.js, partida.js, Persistencia.js

8. Desconexión, reconexión y abandono WebSocket

Una desconexión inesperada activa un período de gracia de 30 s. Reconectar cancela el abandono; salir explícitamente o vencer el timer concreta el abandono con penalización de −50 pts. 

sequenceDiagram
    actor U as Usuario
    participant B as Browser (partida.js)
    participant MM as ManejadorMensajes
    participant PC as PartidaController
    participant MC as ManejadorConexiones
    participant P as Persistencia
    participant Otros as Otros jugadores

    rect rgb(30, 40, 55)
        Note over U,Otros: Desconexión inesperada
        B-xMM: WebSocket close (sin abandonar-partida)
        MM->>PC: desconectar(partidaId, jugadorId)
        PC->>PC: Iniciar timer 30 s (grace period)
        PC->>MC: emitirATodos("jugador-desconectado")
        MC-->>Otros: jugador marcado offline
    end

    rect rgb(30, 40, 55)
        Note over U,Otros: Reconexión exitosa
        B->>MM: WebSocket reconnect /ws?partidaId=X
        MM->>PC: unirJugador(partidaId, jugadorId)
        PC->>PC: Cancelar timer de gracia
        PC->>MC: emitirATodos("jugador-reconectado")
        PC->>MC: emitirA(jugadorId, "estado-partida")
        MC-->>B: Estado completo sincronizado
    end

    rect rgb(30, 40, 55)
        Note over U,Otros: Abandono explícito o gracia vencida
        alt Usuario clic "Salir"
            U->>B: Confirmar abandono
            B->>MM: {accion: abandonar-partida}
            MM->>PC: abandonarPartida(partidaId, jugadorId)
        else Timer 30 s expira
            PC->>PC: concretarAbandono(jugadorId)
        end

        alt En lobby (esperando)
            PC->>PC: Remover jugador de sala
            PC->>MC: emitirATodos("jugador-salio")
            opt Sin humanos restantes
                PC->>P: eliminarPartida(partidaId)
            end
        else Partida en curso
            PC->>P: guardarResultadoPartida (delta −50)
            PC->>P: eliminarPartida(partidaId)
            PC->>MC: emitirATodos("jugador-abandono")
            MC-->>Otros: Partida finalizada por abandono
        end
    end

Archivos: partida.js, PartidaController.js (desconectar, abandonarPartida)
